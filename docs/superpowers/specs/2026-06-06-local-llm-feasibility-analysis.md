## 1. 核心定位

本地大模型的角色是**"化妆品"而非"骨骼"**。

```
游戏骨架（始终运行，不依赖 LLM）：
  ├── GameEngine 所有数值计算
  ├── NpcTraitEvaluator 特质修正
  ├── 好感/皇权/民心/国库 全部由代码计算
  ├── 抄家/赈灾/发饷/卖官 全流程
  ├── 奏折过期/跨级提拔惩罚
  └── ChoiceEngine 场景决策（加权随机）

化妆品（可选，LLM 增强）：
  └── LLM Polish：接收预制文本 → 润色 → 返回
     失败 → 使用原文，游戏正常运行
```

---

## 2. 技术栈

| 组件 | 选型 | 理由 |
|---|---|---|
| 推理引擎 | **LLamaSharp** (llama.cpp C# 封装) | 最成熟的 C# LLM 库，NuGet 直接引用 |
| 模型格式 | **GGUF** Q4_K_M 量化 | llama.cpp 原生格式，千问官方提供 |
| GPU 后端 | **CUDA**（主力）/ **Vulkan**（兜底） | 覆盖 NVIDIA + AMD + Intel |
| 推荐模型 | Qwen2.5-7B-Instruct Q4_K_M | 4.68 GB，中文最优 |
| 最小模型 | Qwen2.5-3B-Instruct Q4_K_M | ~2 GB，Steam Deck 可跑 |

---

## 3. 模型只做一件事：文本润色

### 3.1 输入

```
ChoiceEngine + TextLibrary 产出的预制文本：

  「陛下圣明！冀州大旱三月，赤地千里，
    臣曹操愿领旨开仓赈济！」

  约 50 字，已完成事实填充。
```

### 3.2 输出

```
LLM 润色后：

  「陛下圣明！冀州自入春以来滴雨未降，田间龟裂如网，
    饿殍载道，十室九空。臣曹操虽官微言轻，然见百姓
    流离，五内如焚。愿以项上人头担保，领旨督办赈灾，
    必使粒米归仓，绝不使一钱落入蠹吏之手！」

  约 120 字，增加了细节和角色性格，但未改变任何事实。
```

### 3.3 失败处理

```
LLM 超时（3 秒）/ 返回空 / 格式异常
  → 跳过润色，直接使用输入原文
  → 游戏体验不受任何影响
```

---

## 4. 为什么润色比生成简单得多

| 对比维度 | 全量生成 | 文本润色 |
|---|---|---|
| 任务本质 | 开放式创作 | 受限改写 |
| 输出格式 | 必须结构化（JSON） | 纯文本 |
| 格式错误率 | ~15%（7B） | <1%（7B） |
| 需要理解游戏规则 | ✅ 必须 | ❌ 不需要 |
| 需要保持一致角色 | ✅ 必须跨调用保持 | ❌ 原文已确定角色 |
| 输出长度控制 | 难（容易过长） | 易（原文约 3x 膨胀） |
| prompt 调优难度 | 高 | 低 |
| 对模型能力要求 | 7B 起步 | 3B 即可 |

**润色是一个"安全"的 NLP 任务。** 输入已经包含了所有需要的事实、角色立场、上下文。LLM 只是把 50 字的骨架扩展成 150 字的血肉。

## 5. VRAM 与硬件分析

### 5.1 内存占用

```
Qwen2.5-7B Q4_K_M (4060 8GB)：
  ├── 模型权重         ~4.8 GB
  ├── KV Cache (512)   ~0.4 GB  ← 润色不需要长上下文
  ├── CUDA 运行时      ~0.3 GB
  └── 游戏渲染 (Godot) ~1.2 GB
  ─────────────────────────
  总计：               ~6.7 GB  ✅ 8GB 显存足够

Qwen2.5-3B Q4_K_M (次低配)：
  ├── 模型权重         ~2.0 GB
  ├── 其余开销         ~0.5 GB
  └── 游戏渲染         ~1.2 GB
  ─────────────────────────
  总计：               ~3.7 GB  ✅ 4GB 显卡也能跑
```

### 5.2 延迟

| 硬件 | 模型 | 输入 | 输出 | 预估延迟 |
|---|---|---|---|---|
| 4060 8GB | 7B Q4 | 50 tok | 150 tok | **1-2 秒** |
| 3060 12GB | 7B Q4 | 50 tok | 150 tok | **1-2 秒** |
| 4060 8GB | 3B Q4 | 50 tok | 120 tok | **<1 秒** |
| 1660 6GB | 3B Q4 | 50 tok | 120 tok | **2-3 秒** |
| Steam Deck | 1.5B Q4 | 50 tok | 100 tok | **2-3 秒** |

**所有场景延迟都在可接受范围内。** 大朝会的延迟被三段仪式动画完全掩盖。

### 5.3 并发

本地模式**不需要并发**。游戏是回合制的，每次操作只触发一次 LLM 调用。上一次调用完成后再处理下一次。不存在多请求排队问题。

## 6. 预制文本库规模估算

### 6.1 覆盖场景

| 接口 | 场景类型 | 变体方向 | 模板数 |
|---|---|---|---|
| **大朝会群辩** | 赈灾 / 赏罚 / 军务 | 朝局 3 种（正常/危机/高压） | 15 |
| **旬更情报** | 密报 / 灾情 / 人事 / 边境 / 财政 | 每种 3 个视角 | 15 |
| **奏折生成** | 急报 / 弹劾 / 邀功 / 劝诫 / 建议 | 每种 2-3 模板 | 12 |
| **单大臣对话** | 讨好 / 反对 / 中立 | × 4 个主要大臣 | 36 |
| **随机事件** | 天灾 / 喜讯 / 疾病 | 每种 3 个 | 9 |
| **叙事渲染** | 赈灾总结 / 抄家总结 / 朝会总结 | 每种 2 模板 | 6 |
| **总计** | | | **~93 个模板** |

每个模板 50-200 字（含占位符），**总计约 8000 字**。相当于写一篇公众号文章的工作量。

### 6.2 模板示例

```
TextLibrary[“大朝会_赈灾_危机”] = [
  {
    // 模板A：何进激进 → 曹操谨慎 → 张让反对
    speeches: [
      { minister: “he_jin”, text: “陛下！{灾区}大旱，饿殍遍野。臣请速发国库{amount}万钱赈济!” },
      { minister: “cao_cao”, text: “何将军所言极是，然需防经办之人中饱私囊。” },
      { minister: “zhang_rang”, text: “陛下三思！国库空虚，此时大开国库，西园新军无饷恐生变!” }
    ]
  },
  { /* 模板B：不同角色组合 */ },
  { /* 模板C：不同立场排列 */ }
]
```

## 7. 风险矩阵

| 风险 | 概率 | 影响 | 缓解措施 |
|---|---|---|---|
| LLM 加载失败（驱动不兼容） | 中 | 低 | 自动降级到基础模式 |
| 润色输出语义偏离原文 | 低 | 低 | Prompt 强约束 + 原文已经可用 |
| 模型文件下载失败（Steam） | 低 | 低 | 模型为可选 DLC，不装也能玩 |
| VRAM 不足（后台程序占用） | 中 | 低 | 启动时检测，自动降到 3B 模型或 CPU |
| 首次加载模型 30 秒 | 确定 | 低 | 显示加载动画 "正在准备内阁智囊…" |
| Steam Deck 性能不足 | 中 | 中 | 提供 1.5B 轻量模型，或建议关掉 LLM |
| 语言模型输出不适宜内容 | 极低 | 高 | System prompt 强约束 + 输出黑名单过滤 |

**没有高影响高概率的风险。** 最高的两个风险（VRAM 不足、Steam Deck 性能）都有清晰的降级路径。

## 8. 结论

```
┌──────────────────────────────────────────────────────┐
│                                                    │
│   本地嵌入大模型：✅ 技术上 100% 可行               │
│                                                    │
│   理由：                                            │
│   1. LLM 任务是"润色"不是"生成" → 难度降一个数量级    │
│   2. LLM 失败 → 直接跳过，游戏零影响                  │
│   3. 3B 模型即可胜任 → VRAM 需求低至 2GB             │
│   4. 延迟 1-2 秒 → 被游戏动画掩盖                    │
│   5. 代码结构简单 → LLamaSharp 30 行 + 业务胶水 ~100 行│
│                                                    │
│   最大工程量：写 ~93 个预制文本模板（约 8000 字）      │
│   次大工程量：调 prompt（10-20 轮迭代）               │
│                                                    │
│   与云端模式的差异：                                  │
│   · 本地模式 = 确定性游戏 + 文本润色                   │
│   · 云端模式 = 开放性游戏 + 全量生成                   │
│   · 两者共用同一套接口，用户随时切换                   │
│                                                    │
└──────────────────────────────────────────────────────┘
```
