# NPC 年龄/传记核查与关系网功能可行性分析

> 状态：📋 分析阶段 — 尚未实施代码修改
>
> 本文档用于在落地代码前核查当前 71 名 NPC 的年龄/寿命/史料定位，并分析“关系网”系统的可行性、数据模型、规则接入点与 UI 分期方案。

## 一、当前实现概况

- NPC 预设文件：`donghan/Backend/DonghanEngine.Core/HistoricalNpcPresets.cs`
- 开局部署列表：`donghan/Backend/DonghanEngine.Core/GameState.cs`
- 当前预设池：71 人
- 当前开局可见朝臣：27 人
- 生命周期使用字段：
  - `BirthYear`：用于 `state.Year - npc.BirthYear` 动态计算年龄。
  - `BaseLongevity`：不是“卒年”，而是期望寿命；当年龄超过该值后，每年 15% 概率寿终。
  - 因此如果要强历史节奏，可令 `BirthYear + BaseLongevity` 接近史实卒年；如果要允许架空，则 `BaseLongevity` 可以作为软上限。

## 二、年龄/传记核查结论

### 2.1 重要发现

当前 `BirthYear` 与 `BaseLongevity` 混用了两种意图：

1. 对有明确生卒年的名臣，部分字段接近真实年龄/卒年。
2. 对生年不详人物，使用了“让 184 年年龄合理”的估算。
3. 对部分史实卒年很近的人物，`BaseLongevity` 过高，会让其大概率活到与史实明显不符的时间点。

游戏是架空改写，允许不完全按史实死亡，但如果用户希望“历史人物年龄与传记更可信”，建议：

- `BirthYear`：尽量用明确生年；无明确生年则估算，并在 `HistoricalRole` 或未来 `HistoricalNotes` 标明“生年不详，按中年/青年估”。
- `BaseLongevity`：改为“历史倾向寿数”，让 `BirthYear + BaseLongevity` 接近史实卒年；再通过游戏事件允许改写。
- 新增字段可选：`HistoricalDeathYear`、`SourceNote`，避免把“寿命上限”和“史实卒年”混在一起。

### 2.2 P0 — 会明显误导史实或游戏节奏的问题

#### P0-1：卢植出生年当前为 139，导致 184 年 45 岁，偏年轻

- 当前：`lu_zhi` BirthYear=139, BaseLongevity=53 → 约 192 卒
- 史实：卢植生年不详，多作“2世纪？—192”；不宜写成太年轻。
- 问题：刘备师长、经学大儒、黄巾前线统帅，在 184 年更应呈现老成名宿，而非 45 岁中年。
- 建议：BirthYear 改为约 125–135；BaseLongevity 调整为约 57–67，使预期退场仍接近 192。

#### P0-2：皇甫嵩当前预期 190 年退场，早于史实 195

- 当前：BirthYear=130, BaseLongevity=60 → 约 190
- 史实：皇甫嵩卒于 195，生年不详。
- 问题：黄巾后续、董卓前后军方压舱人物会过早自然退场。
- 建议：BaseLongevity 改为 65 左右。

#### P0-3：朱儁当前出生年 135，184 年 49 岁，且预期 195 退场刚好；但史料通常生年不详

- 当前：BirthYear=135, BaseLongevity=60 → 195
- 史实：朱儁卒于 195，生年不详。
- 问题：数值上可用，但“出生年确定”会造成伪精确。
- 建议：若保留当前值，应在注释/SourceNote 说明“生年估算”；若增加 `HistoricalBirthYearApprox` 更佳。

#### P0-4：桥玄应为 110—184，而当前 109—约185

- 当前：BirthYear=109, BaseLongevity=76 → 185
- 史实：桥玄 110—184。
- 问题：开局 184 年桥玄已是垂暮老臣，且史实当年卒；现在会多活一点但问题不大。
- 建议：BirthYear=110, BaseLongevity=74 或 75。若要保留开局登场，可作为“病重老臣/清议压舱石”，死亡事件概率较高。

## 三、建议的数据结构修正

### 3.1 最小修正方案

不新增字段，只修 `BirthYear` / `BaseLongevity` / `HistoricalRole`：

- 优点：改动小，测试少，马上能改善可信度。
- 缺点：无法区分“史实卒年”和“游戏寿命软上限”；未来还会争论为什么某人没按史实死。

### 3.2 推荐方案

在 `NpcState` 新增：

```csharp
public int? HistoricalDeathYear { get; set; } = null;
public string SourceNote { get; set; } = string.Empty;
```

含义：

- `HistoricalDeathYear`：史实或传统说法的卒年；不强制死亡，只用于事件/档案/关系网。
- `SourceNote`：如“《后汉书》有传，生年不详；出生年为游戏估算”。
- `BaseLongevity`：仍作为游戏生命周期软上限。

这样以后可以做：

- 黄巾/宫变/董卓入京等历史节点触发“史实倾向退场”。
- 玩家改写历史时，可以让人物避开史实死亡。
- 档案页可显示“史实卒年：189（可因玩家干预改写）”。

## 四、关系网功能可行性分析

### 4.1 结论

可行，而且很适合当前项目。原因：

- 当前 NPC 已有 `Faction`、`Power`、`Favorability`、`Ambition`、`Traits`、`InitialLocation`、`HistoricalRole`，关系网可以在这些字段上自然扩展。
- 游戏已有强 NPC 博弈场景：朝会、抄家、地方任命、平叛、招安、西园、奏折批阅。
- 玩家已经需要理解“谁是谁的党羽/师友/宗族/敌人”，关系网能把政治风险可视化，提升策略深度。

推荐先做“半静态关系网 + 少量动态强度变化”，不要一开始做完整图数据库或复杂 AI 社交模拟。

### 4.2 核心设计目标

关系网不只是百科展示，而要服务玩法：

1. 让玩家看懂朝局：外戚、阉党、清流、袁氏、颍川士人、凉州边军、刘备义从等网络。
2. 让行动有连锁反应：抄家、提拔、外放、平叛、奏折裁断会影响相关人物。
3. 让高能力人物不再是免费答案：重用袁绍会增强袁氏门生；外放董卓/公孙度会带来边镇坐大；清流互保会降低暴力抄家的收益。
4. 让 UI 更有历史感：大臣档案可以显示“宗族/师友/党羽/宿敌/门生故吏”。

### 4.3 数据模型建议

新增关系类型枚举：

```csharp
public enum NpcRelationType
{
    Kinship,        // 宗族/姻亲
    Patronage,      // 举主/门生故吏/提携
    FactionAlly,    // 同党/同派
    TeacherStudent, // 师生
    SwornBond,      // 义从/结义/亲密战友
    Rivalry,        // 政敌
    Hostility,      // 敌对/仇怨
    Command,        // 上下级/部曲
    RegionalTie     // 同乡/同郡/州郡网络
}
```

新增关系边：

```csharp
public class NpcRelation
{
    public string FromNpcId { get; set; } = string.Empty;
    public string ToNpcId { get; set; } = string.Empty;
    public NpcRelationType Type { get; set; }
    public int Strength { get; set; } = 50; // 0-100
    public bool IsMutual { get; set; } = true;
    public string Label { get; set; } = string.Empty;
    public string HistoricalBasis { get; set; } = string.Empty;
}
```

`GameState` 增加：

```csharp
public List<NpcRelation> NpcRelations { get; set; } = new();
```

并建静态预设：

```csharp
public static class HistoricalNpcRelations
{
    public static List<NpcRelation> All { get; } = new() { ... };
}
```

### 4.5 规则接入点

#### 抄家/籍没

当前 `FindConfiscationFramer` 只看“好感 >=55 且腐败 <40”。关系网可增强：

- 若目标有强盟友/宗族，抄家后盟友好感下降、权势反扑。
- 若弹劾者与目标有 Rivalry/Hostility，弹劾更容易成功，但可能被认为党争。
- 若目标是清流网络核心，民心/皇权反噬更强。

#### 任命地方官

- 外放某人会增强其 RegionalTie / Command 关系。
- 同党人物可能要求援引门生故吏，形成“地方小集团”。
- 高野心 + 强关系网 + 地方低民心 = 更高叛乱/割据概率。

#### 朝会议题

- 群臣发言不再只按 `Faction`，而按关系扩散：
  - 袁绍被提拔，袁氏相关人物支持，阉党反对。
  - 抑制宦官，十常侍网络集体辩护，何进/袁绍/清流支持。
- UI 可在发言下显示“受袁氏宗族影响”“受十常侍同党影响”。

#### 生命周期/事件触发

- 部署刘备时可自动提示“关羽、张飞可随义兵触发”。
- 部署董卓时可逐步引出李傕、郭汜、张济、贾诩。
- 宫变事件可用关系网判定何进/十常侍/袁绍/蹇硕的阵营。

### 4.6 UI 入口方案

#### MVP：大臣档案内的“关系札记”

最小可见改动：在 `BuildMinisterDossierText` 增加：

```text
【关系札记】
宗族：袁隗（宗族长辈，强度85）
同党：袁术（袁氏门阀，强度65）
政敌：张让（诛宦对立，强度80）
```

优点：开发量小，不需要复杂图形布局。

#### 第二阶段：关系网弹窗

在大臣档案底部加“查看关系网”按钮，弹出 `PopupSkin.Document` 或新 `PopupSkin.Network`：

- 左栏：中心人物卡
- 中栏：关系分组列表（宗族/师友/同党/政敌）
- 右栏：选中关系详情与玩法影响

不建议第一阶段做自由拖拽节点图，因为 Godot 动态图布局、缩放、连线交互会显著增加工作量。

#### 第三阶段：朝局总览关系图

在“朝会”或“黄门密札”增加“朝局关系”页：

- 以派系为簇：外戚、阉党、清流、袁氏、边军、地方州牧。
- 只显示 active NPC，冷备关系可灰显。
- 支持筛选“同盟/敌对/宗族/师生”。

### 4.7 分期实施建议

#### Phase 1：静态关系数据 + 档案展示

- 新增 `NpcRelationType`、`NpcRelation`、`HistoricalNpcRelations`。
- `GameState` 初始化关系表。
- `BuildMinisterDossierText` 增加“关系札记”。
- 测试：关系 ID 存在、无重复、引用 NPC 在预设池或开局池中存在、关键关系存在。

预计风险低，适合下一步做。

#### Phase 2：关系影响规则

- 抄家反噬接入关系网。
- 提拔/贬斥/外放影响相关人物好感。
- 朝会议题发言倾向受关系修正。

预计风险中等，需要更多测试。

#### Phase 3：关系网专属弹窗/总览

- 先列表式关系网。
- 再考虑可视化节点图。
- 需要 Godot headless + 截图验证，避免动态 UI 重叠。

预计风险中高，适合在数据和规则稳定后做。

## 五、推荐下一步

建议不要马上一次性修 71 人并做关系网大功能。推荐顺序：

1. 先落地 **年龄/传记修正最小补丁**：修 P0/P1 的 `BirthYear`、`BaseLongevity`、`HistoricalRole` 表述，尤其卢植、皇甫嵩、桥玄、太史慈、十常侍/宫变节点。
2. 再做 **关系网 Phase 1**：静态关系数据 + 大臣档案“关系札记”。
3. 通过测试与 build 后 push。
4. 后续再接入抄家/任命/朝会规则影响。
