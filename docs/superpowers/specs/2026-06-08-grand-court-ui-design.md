## 1. 设计结论

朝会界面不再是一个“输入圣旨”的小弹窗，而应成为玩家临朝听政、观察派系、听取奏对、作出圣裁的核心决策场景。

它承担三件事：

1. **把 NPC 从主界面移入具体场景**：主界面不常驻 NPC；朝会中才显示在朝百官、派系站位与发言。
2. **把自由输入变成补充能力**：默认给玩家清晰议题和裁断按钮；“亲拟圣旨”保留开放玩法。
3. **把数据交给情报面板**：朝会只显示议题摘要、派系态度、行动后果预估，不展开完整天下数据。

一句话目标：**主界面像御案，朝会像真正的朝堂。**

---

## 2. 非目标

本设计第一轮不做以下内容：

- 不把黄门密札的州郡详情搬到朝会。
- 不在朝会中常驻显示皇权、国库、民心等完整数值表。
- 不要求立刻接入真实 LLM 群臣辩论。
- 不一次性实现所有后端裁断规则。
- 不做复杂 2D 人物立绘站位系统。
- 不把朝会做成独立 Godot Scene；第一版仍可放在 `MainScene.CourtAndOpening.cs` partial 中，后续再拆文件。

## 3. 入口规则

### 3.1 主界面入口

主界面顶部卷轴按钮：

```text
📜 朝会
```

点击后进入朝会逻辑。

### 3.2 场景限制

朝会只能在 `宣政殿` 开启。

如果玩家当前不在宣政殿，不再只显示一句“不可轻动”，而弹出小确认：

```text
【黄门急奏】
陛下尚未驾临宣政殿，百官不可无端入朝。

[起驾宣政殿] [取消]
```

- 点击“起驾宣政殿”：调用 `DoTravel("宣政殿")`，抵达后可再次点朝会。
- 点击“取消”：关闭提示。

第一版为了降低改动，也可以先保留原提示，但设计目标是上述确认框。

## 4. 朝会整体布局

朝会使用大弹窗或近全屏不透明面板，推荐 1100×640 左右。

```text
┌──────────────────────────────────────────────────────────────┐
│ 宣政殿 · 大朝会                         光和7年 · 184年1月上旬 │
├───────────────┬──────────────────────────────┬───────────────┤
│ 百官班列       │ 今日廷议                      │ 御前裁断       │
│               │                              │               │
│ 外戚武臣       │ 【当前议题】整军备寇            │ [准何进整军]   │
│ 何进  请战     │                              │ [命曹操备兵]   │
│               │ 何进：臣请整北军……             │ [令张让筹饷]   │
│ 中官近侍       │ 张让：国帑不可轻动……           │ [转情报详查]   │
│ 张让  观望     │ 曹操：兵贵神速……               │ [亲拟圣旨]     │
│               │                              │               │
│ 西园武臣       │                              │ [退朝]         │
│ 曹操  请任     │                              │               │
│ 蹇硕  主军     │                              │               │
└───────────────┴──────────────────────────────┴───────────────┘
```

### 4.1 顶部标题区

显示：

- `宣政殿 · 大朝会`
- 当前时间：`光和7年 · 184年1月上旬`
- 当前阶段：`择议 / 群辩 / 圣裁 / 纪要`

示例：

```text
宣政殿 · 大朝会
光和7年 · 184年1月上旬  ·  阶段：群臣奏对
```

### 4.2 左栏：百官班列

左栏只显示朝会相关 NPC，不再作为主界面常驻列表。

分组建议：

- 外戚武臣
- 中官近侍
- 西园武臣
- 清流士人（后续 NPC 上线后）
- 地方/外任来使（后续）

第一版根据已有 4 个 NPC：

```text
【外戚武臣】
何进  大将军  请战

【中官近侍】
张让  中常侍  观望

【西园武臣】
曹操  典军校尉  请任
蹇硕  上军校尉  主军
```

显示原则：

- 显示姓名、官职/身份、朝会态度。
- 不显示好感、权势、腐败等硬数值。
- 点击 NPC 名牌可打开既有 `ShowMinisterDetails(npcId)` 详情弹窗。
- 死亡/下野 NPC 不显示。
- 已外任太守 NPC 可显示为灰色：`【外任冀州】`，默认不参与普通朝议。

### 4.3 中栏：今日廷议

中栏是朝会核心，分两个状态。

#### 状态 A：择议

玩家刚打开朝会时显示“今日可议”。

议题卡片示例：

```text
【急奏】冀州黄巾扰动
叛军势大，地方民心低落。建议转黄门密札详查或命将出征。

【常议】整军备寇
北军、西园军久未整备，朝廷武备不足。

【常议】国帑筹措
国库支出渐重，朝臣请议财计。

【自由】亲拟圣旨
陛下可直接口召百官，乾纲独断。
```

第一版固定议题：

1. `整军备寇`
2. `国帑筹措`
3. `整饬宦官`
4. `举荐将才`
5. `亲拟圣旨`

后续动态议题从 `GameState` 生成：

- 有叛乱省份 → `某州黄巾扰动`
- 国库低于阈值 → `国帑告急`
- 西园军低于编制 → `西园军补军`
- 有地方官空缺 → `某州太守空缺`
- 健康过低 → `龙体违和，是否暂缓朝政`

#### 状态 B：群辩/纪要

选择议题后，中栏显示群臣发言。

格式：

```text
【议题】整军备寇

何进：
“黄巾虽乱，朝廷威灵尚在。臣请整北军，明示天下。”
倾向：请战 / 扩权

张让：
“大将军动兵，所费国帑不可不察。奴才以为，应先核军费。”
倾向：制衡外戚 / 掌财

曹操：
“兵贵神速。若迟疑数旬，贼势恐连动并州。”
倾向：请任 / 速战
```

群辩文案第一版可硬编码模板，后续接 `IAIScheduler.OrchestrateGrandCourtAsync(...)`。

### 4.4 右栏：御前裁断

右栏显示当前议题可选裁断。

按钮文案要像圣裁，不像普通菜单。

示例：整军备寇

```text
【御前裁断】
[准何进整北军]
[命曹操整西园军]
[令张让核军费]
[转黄门密札详查]
[亲拟圣旨]
[退朝]
```

示例：国帑筹措

```text
[令张让筹措内帑]
[准西园卖官筹资]
[削减军费支出]
[亲拟圣旨]
[退朝]
```

示例：整饬宦官

```text
[训诫张让]
[命何进约束外廷]
[暂不激化]
[亲拟圣旨]
[退朝]
```

### 4.5 底部：亲拟圣旨抽屉

“亲拟圣旨”不常驻输入框，而是点击后展开。

```text
【亲拟圣旨】
[输入框：如“重赏曹操，命其整备西园军”]
[宣旨起大朝仪] [收起]
```

提交后沿用现有逻辑：

- 运行三段朝仪遮罩。
- 调用 `_gameEngine.ProcessPlayerTurnAsync(text)`。
- 把结果写入主界面 `StoryOutput` 或朝会纪要区。
- `UpdateUI()`。

## 5. 朝会交互流程

### 5.1 打开朝会

1. 玩家点击主界面 `📜 朝会`。
2. 若不在宣政殿：提示起驾。
3. 若在宣政殿：打开朝会面板。
4. 朝会默认进入“择议”状态。

### 5.2 选择议题

1. 玩家点击一个议题卡。
2. 中栏刷新为该议题的群臣奏对。
3. 右栏刷新为该议题的裁断按钮。
4. 左栏 NPC 态度随议题刷新。

### 5.3 玩家圣裁

裁断分三类：

#### A. 规则裁断

能直接对应现有后端方法的动作。

例：

- 转黄门密札详查：关闭朝会，打开情报面板。
- 命曹操整西园军：如果当前设计尚无后端规则，先转为叙事/自由输入。
- 训诫张让：可走 `ProcessPlayerTurnAsync("训诫张让")`。

#### B. 导航裁断

引导玩家去另一个面板完成具体操作。

例：

- `转黄门密札详查` → 打开 `IntelPanel`
- `前往西园整军` → 起驾 `西园`

#### C. 亲拟圣旨

使用自由输入。

### 5.4 退朝

点击“退朝”：

- 关闭朝会面板。
- 主界面故事区可显示一条短文本：

```text
【退朝】
静鞭三响，百官次第退出宣政殿。陛下暂收朝议，天下仍在暗流中流转。
```

## 6. 议题系统设计

### 6.1 数据模型建议

第一版可以先在前端定义简单内部类，不急着进后端：

```csharp
private sealed class CourtTopicViewModel
{
    public string Id { get; init; } = string.Empty;
    public string Category { get; init; } = string.Empty; // 急奏 / 常议 / 自由
    public string Title { get; init; } = string.Empty;
    public string Summary { get; init; } = string.Empty;
    public List<CourtSpeechViewModel> Speeches { get; init; } = new();
    public List<CourtDecisionViewModel> Decisions { get; init; } = new();
}

private sealed class CourtSpeechViewModel
{
    public string MinisterId { get; init; } = string.Empty;
    public string MinisterName { get; init; } = string.Empty;
    public string Faction { get; init; } = string.Empty;
    public string Speech { get; init; } = string.Empty;
    public string Attitude { get; init; } = string.Empty;
}

private sealed class CourtDecisionViewModel
{
    public string Id { get; init; } = string.Empty;
    public string Label { get; init; } = string.Empty;
    public string Hint { get; init; } = string.Empty;
}
```

### 6.2 第一版固定议题

#### 议题一：整军备寇

- 何进：请求整北军。
- 张让：担心军费与外戚坐大。
- 曹操：主张速整西园军。
- 蹇硕：请求扩充西园军。

裁断：

- `准何进整北军` → 暂走自由输入：`重赏何进，命其整军备寇`
- `命曹操整西园军` → 若在宣政殿，提示可起驾西园；后续接阅兵/募兵。
- `令张让核军费` → 暂走自由输入：`命张让核查军费`
- `转黄门密札详查` → 打开情报面板。

#### 议题二：国帑筹措

- 张让：请以内廷渠道筹措。
- 何进：反对宦官染指军费。
- 曹操：提醒军费不可断。

裁断：

- `令张让筹措内帑` → 自由输入：`命张让筹措国帑`
- `前往西园筹资` → 起驾西园。
- `暂缓筹措` → 退回择议。

#### 议题三：整饬宦官

- 何进：请求裁抑中官。
- 张让：辩称内廷忠心。
- 曹操：建议不可骤激。

裁断：

- `训诫张让` → 现有 Mock 支持关键词，可调用 `ProcessPlayerTurnAsync("训诫张让")`。
- `安抚张让` → `ProcessPlayerTurnAsync("重赏张让")`（后续补 Mock）。
- `暂不激化` → 退回择议。

#### 议题四：举荐将才

- 何进：举荐外戚军中将佐。
- 曹操：自请任事。
- 蹇硕：主张西园诸校尉可用。

裁断：

- `召见曹操` → 打开 NPC 详情。
- `召见蹇硕` → 打开 NPC 详情。
- `转黄门密札任官` → 打开情报面板。

#### 议题五：亲拟圣旨

直接展开输入框。

## 7. 视觉风格

### 7.1 色彩

- 背景：深漆木黑褐 `#15110D`
- 边框：暗金 `#B9962E`
- 朝臣卡：深红/深褐底
- 御批按钮：朱红边或朱砂 hover
- 急奏标签：暗红
- 常议标签：暗金
- 自由标签：青灰

### 7.2 控件拟物

- 议题卡：折叠奏疏/笏板样式。
- 发言卡：展开黄绢/奏对记录。
- 裁断按钮：朱批御印样式。
- NPC 名牌：朝臣站班木牌。

### 7.3 信息密度

朝会不是情报面板，单屏只应出现：

- 当前议题 1 个。
- 朝臣 4–8 个。
- 发言 3–5 条。
- 裁断按钮 3–6 个。

超过这些应滚动，不扩大常驻信息。

### 8.2 `IAIScheduler`

现有接口：

```csharp
Task<AIOrchestrationResult> OrchestrateGrandCourtAsync(string playerInput, string activeOfficerId, GameState state);
```

后续可用于：

- 玩家选择议题后，把议题转为 `playerInput`。
- `activeOfficerId` 第一版可为空或使用主导大臣 ID。

### 8.3 `ProcessPlayerTurnAsync`

现有自由输入仍使用它。

建议所有第一版裁断在尚无明确规则时，都可转为一句自然语言输入：

```csharp
await RunCourtCommandAsync("训诫张让");
await RunCourtCommandAsync("重赏何进，命其整军备寇");
```

这样 UI 先闭环，不阻塞后端规则扩展。

### 8.4 黄门密札

朝会中的“详查”“任官”“平叛”按钮应跳转情报面板，而不是在朝会里复制情报系统。

## 9. 第一轮实现建议

### 第一轮目标

只做 UI 骨架和交互闭环，不做新后端规则。

完成后玩家体验应是：

1. 点击主界面 `📜 朝会`。
2. 进入大朝会三栏界面。
3. 看到百官班列、议题列表、裁断区。
4. 点击议题后看到群臣发言。
5. 点击裁断按钮后能触发已有流程或跳转其他面板。
6. 点击亲拟圣旨可继续使用原自由输入。

### 第一轮文件

- 修改：`donghan/Frontend/MainScene.CourtAndOpening.cs`
- 可能修改：`donghan/Frontend/MainScene.Style.cs`
- 不修改后端。
- 不改测试数量，主要用前端 build 验证。

### 第一轮任务拆分

1. 将 `_courtPopup` 改为 1100×640 大面板。
2. 新增字段：左栏容器、中栏容器、右栏容器、亲拟圣旨输入区。
3. 提取 `BuildCourtHeader()`。
4. 提取 `BuildCourtOfficialsColumn()`。
5. 提取 `BuildCourtTopicsColumn()`。
6. 提取 `BuildCourtDecisionColumn()`。
7. 新增固定 `CourtTopicViewModel` 列表。
8. 点击议题时刷新发言与裁断。
9. 点击亲拟圣旨展开输入区，复用 `OnConfirmCourtAssembly()`。

## 10. 后续增强路线

### 第二轮：动态议题

从 `GameState` 自动生成议题：

- 叛乱省份。
- 低民心省份。
- 国库告急。
- 西园军缺编。
- 太守空缺。

### 第三轮：真实群辩

接入 `IAIScheduler.OrchestrateGrandCourtAsync`：

- 议题选择后异步生成群臣发言。
- 朝仪遮罩期间后台请求。
- 请求失败时回退硬编码模板。

### 第四轮：圣裁规则化

把常见裁断从自然语言输入升级为明确后端方法：

- 朝会训诫/赏赐。
- 朝会授权军费。
- 朝会指定军务主导人。
- 朝会影响派系权势。

### 第五轮：朝会纪要

每次朝会生成纪要写入 Chronicle：

```text
光和七年正月上旬，大朝会，议整军备寇。帝命曹操整西园军。
```

## 11. 验收标准

设计实现后应满足：

- 主界面仍保持极简，不恢复 NPC 或硬数值常驻。
- 朝会面板打开后能清楚看到“百官班列 / 今日廷议 / 御前裁断”。
- 玩家不用自由输入也能完成至少 3 个朝会议题的选择与裁断。
- 自由输入仍可用，不破坏原 `ProcessPlayerTurnAsync`。
- 朝会不复制黄门密札的详细数据，只提供跳转。
- 前端 `dotnet build DonghanFrontend.csproj -v minimal` 通过。
- 后端测试仍通过。

## 12. 风险与取舍

### 风险一：硬编码议题显得假

第一版允许硬编码，因为目标是完成 UI 体验闭环。第二轮再动态化。

### 风险二：朝会变成另一个信息大屏

通过限制每屏信息量解决：不显示完整数值表，不展开州郡详情。

### 风险三：裁断没有后端规则

第一版用自然语言输入或导航动作兜底。等 UI 跑通后再逐步规则化。

### 风险四：文件继续变长

若 `MainScene.CourtAndOpening.cs` 超过 500 行，应拆出 `MainScene.CourtPanel.cs`，但第一轮不必为拆而拆。
